# 浏览器调试工具差异

## 背景

在调试网页时，发现 `playwright-cli` 和 `Chrome DevTools MCP` 对网络请求的监控结果不一致。

## 问题描述

使用 `playwright-cli network` 命令监控网络请求时，点击分类后没有捕获到 API 请求；但使用 `Chrome DevTools MCP list_network_requests` 却能正确看到 API 请求。

## 原因分析

### 1. 监控时机不同

| 工具 | 行为 |
|------|------|
| Playwright CLI `network` | 需要在操作**之前**就启用监控，它读取的是日志缓冲区 |
| Chrome DevTools MCP | 实时监控，直接连接到浏览器，能立即看到新请求 |

### 2. 浏览器实例不同

| 工具 | 行为 |
|------|------|
| Playwright CLI | 打开新的、隔离的浏览器实例，可能有缓存或状态差异 |
| Chrome DevTools MCP | 直接连接到用户已打开的浏览器，使用完整状态 |

### 3. Network API 实现差异

- **Chrome DevTools Protocol (CDP)** 的网络监控是**实时推送**的
- **Playwright CLI** 的 `network` 命令可能只是**轮询**读取已有日志

## 正确用法

### Playwright CLI

```bash
# 在点击之前就获取初始网络状态
playwright-cli network  # 第一次调用清空/记录初始状态
# 然后执行点击操作
playwright-cli click <element>
# 立即再次获取网络状态
playwright-cli network
```

### Chrome DevTools MCP (推荐)

```typescript
// 实时监控，直接获取所有请求
mcp__chrome-devtools__list_network_requests()
// 点击操作
mcp__chrome-devtools__click(uid)
// 再次获取，能看到新增的请求
mcp__chrome-devtools__list_network_requests()
```

## 结论

| 场景 | 推荐工具 |
|------|----------|
| 实时交互调试 | **Chrome DevTools MCP**（实时、准确） |
| 自动化测试 | Playwright CLI（适合无头测试） |
| 截图/UI验证 | Playwright CLI |

**调试网络请求时，优先使用 Chrome DevTools MCP**。
