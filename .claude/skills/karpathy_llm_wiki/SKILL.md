# LLM Wiki 知识库构建技能

基于 Karpathy 提出的 LLM Wiki 方案，将知识管理从"临时搜索"升级为"永久复利"。

## 核心思想

| 传统方式 | LLM Wiki 方式 |
|----------|---------------|
| 每次从零开始 | 编译一次，持续复用 |
| 碎片化无关联 | 自动建立链接 |
| 临时检索 | 永久沉淀 |

## LLM Wiki 三层架构

```
├── raw/      # 原始素材（只读，Source of Truth）
├── wiki/     # LLM 生成的知识库（LLM 独占领地）
└── output/   # 查询产物（可回流到 wiki 层）
```

## 三条铁律

1. **raw 不可变**：原始资料是唯一的真实来源，LLM 只读不写
2. **wiki LLM 独占**：所有 Wiki 内容由 LLM 编写和维护，人不直接编辑
3. **好的 output 要回流**：有价值的查询结果归档回 Wiki，形成复利

## 工作流程

### 第一步：分析目录结构

首先检查目标目录是否已有 LLM Wiki 结构：

```
输入：目录路径（用户通过 skill 参数传入）
输出：目录结构分析报告
```

如果目录不存在，询问用户是否要创建新结构。

### 第二步：阅读 raw/ 原始笔记

1. 扫描 `raw/` 目录下的所有文件
2. 按类型分组（会议纪要、面试记录、工作笔记等）
3. 阅读并理解每个文件的内容
4. 生成内容摘要

### 第三步：在 wiki/ 生成结构化知识

**关键原则：wiki/ 的子目录结构应该动态适应 raw/ 的实际结构，不要硬编码固定目录名。**

1. **先分析 raw/ 的目录结构**
   - 扫描 `raw/` 下有哪些子目录和文件
   - 根据原始内容的性质（会议纪要、面试记录、项目文档等）确定 wiki/ 的目录结构
   - wiki/ 的子目录可以与 raw/ 的子目录一一对应，也可以重新组织

2. **创建对应的 wiki/ 结构**
   - 如果 `raw/会议纪要/` 存在，创建 `wiki/会议纪要/`
   - 如果 `raw/面试记录/` 存在，创建 `wiki/人员档案/`（语义映射）
   - 如果 `raw/项目A/` 存在，创建 `wiki/项目A/`
   - 如果有散文件，按内容性质归类到合适的目录

3. **为每个 raw/ 文件生成对应的 wiki 页面**
   - 保持与 raw/ 文件的对应关系
   - 在 wiki 页面顶部标注对应的源文件（反向链接到 raw/）
   - 使用 wiki-links 互相引用

每篇 wiki 文章应该：
- 使用中文
- 通过 wiki-links 相互引用
- 包含 Mermaid 图表（如果适合）
- 有清晰的层次结构

### 第四步：归档有价值的 output

如果用户在 wiki/ 目录下进行了查询或分析：
1. 评估输出是否有价值归档
2. 如果有价值，生成结构化文档存入 wiki/ 对应子目录
3. 更新 index.md 的关联链接

## 输出格式

### wiki/index.md 示例

```markdown
# [主题名] Wiki 索引

## 目录结构
本 wiki 镜像 raw/ 的结构，每个目录对应原始素材：

- [[会议纪要/]] - 会议讨论记录
- [[人员档案/]] - 人物相关信息（源自面试记录等）
- [[项目/]] - 项目相关文档

## 最近更新
> YYYY-MM-DD - 更新内容简述
```

### wiki/[对应raw目录]/[页面名].md 示例

```markdown
# 页面标题

> 源文件：[[../../raw/会议纪要/原始文件名]]

## 概述
一句话描述。

## 详情
详细内容...

## 相关链接
- [[../其他相关页面]]
```

## 使用方式

用户在调用 skill 时通过参数指定目录路径，例如：
```
/karpathy_llm_wiki 对 datatist 目录执行 LLM Wiki 整理
```

或者直接说：
```
帮我整理一下这个目录的知识库结构
```

## 注意事项

- raw/ 下的文件永远不要修改，只读
- wiki/ 内容完全由 LLM 生成和维护
- 保持 wiki-links 的双向链接
- 定期进行"健康检查"确保知识一致性
